This file defines the expected enterprise standard for selecting and designing storage in a modular RAG system.

Storage affects:

- ingestion
- indexing
- retrieval
- metadata filtering

- persistence
- scalability
- cost
- observability

- reindexing

Do not assume a vector database unless the user explicitly selects one or allows a default.

Storage must be chosen based on the user's runtime, corpus size, retrieval strategy, persistence needs, security model, and operational constraints.

A RAG system may need to store:

- raw documents

- chunks

- cache entries

- audit records

- where did this result come from?

## Storage Decision Factors

Before selecting storage, determine the following.

### 1. Corpus Size

Examples:

- fewer than 1,000 chunks
- 1,000 to 100,000 chunks
- 100,000 to 1,000,000 chunks
- millions of chunks

Why it matters:

- small corpora can use in-memory, JSON, or SQLite
- medium corpora often need SQLite, PostgreSQL, or vector indexes
- large corpora often need PostgreSQL with pgvector or a dedicated vector database

### 2. Persistence Requirement

Examples:

- temporary in-memory only
- local persistent files
- local database
- production persistent index
- distributed persistent storage

Why it matters:

- prototypes can rebuild indexes on restart
- production systems usually cannot
- serverless runtimes may not support durable local disk

### 3. Retrieval Strategies

Examples:

- keyword only
- embedding only
- hybrid keyword plus embedding
- contextualized hybrid
- metadata filtering
- permission-scoped retrieval
- reranking

Why it matters:

- keyword indexes and vector indexes have different storage needs
- hybrid retrieval often needs both lexical and vector storage
- contextual retrieval requires storing original and contextualized text separately

### 4. Runtime and Deployment

Examples:

- local script
- FastAPI app
- Flask app
- Django app
- Node.js service
- Next.js app
- serverless function
- background worker
- enterprise backend

Why it matters:

- local scripts can use files or SQLite
- web apps often need shared persistent storage
- serverless systems need external storage
- enterprise systems often need audit logs and permission enforcement

### 5. Security Scope

Examples:

- personal local corpus
- single-user app
- team workspace
- multi-tenant SaaS
- organization-scoped enterprise search
- permission-scoped document retrieval

Why it matters:

- permission metadata must survive every storage layer
- tenant data may need namespace isolation
- sensitive systems should not rely only on final client-side filtering

### 6. Update Pattern

Examples:

- static corpus
- occasional uploads
- frequent document updates
- real-time ingestion
- deletion and reindexing required
- legal deletion requirements

Why it matters:

- static corpora can rebuild indexes cheaply
- frequently changing corpora need update and invalidation logic
- deletion requires removing chunks, embeddings, keyword entries, contextualized text, and caches

### 7. Operational Complexity

Examples:

- no external services
- managed vector database acceptable
- existing PostgreSQL preferred
- cloud-native architecture required
- local-only required

Why it matters:

- simple storage lowers setup burden
- managed storage improves scale but adds cost and vendor dependency
- enterprise teams often prefer existing infrastructure

## In-Memory Store Example

In-memory storage is useful for tests and examples.

    class InMemoryChunkStore:
        def __init__(self) -> None:
            self._chunks: dict[str, Any] = {}

        def save_chunks(self, chunks: list[Any]) -> None:
            for chunk in chunks:
                if not chunk.chunk_id:
                    raise ValueError("chunk_id cannot be empty.")

                self._chunks[chunk.chunk_id] = chunk

        def load_chunks(self, *, filters: dict[str, Any] | None = None) -> list[Any]:
            active_filters = filters or {}

            chunks = list(self._chunks.values())

            if not active_filters:
                return chunks

            return [
                chunk
                for chunk in chunks
                if metadata_matches_filters(chunk.metadata, active_filters)
            ]

        def get_chunk(self, *, chunk_id: str) -> Any | None:
            return self._chunks.get(chunk_id)

        def delete_chunks_for_document(self, *, document_id: str) -> None:
            chunk_ids_to_delete = [
                chunk_id
                for chunk_id, chunk in self._chunks.items()
                if chunk.document_id == document_id
            ]

            for chunk_id in chunk_ids_to_delete:
                del self._chunks[chunk_id]

Rules:

- good for tests
- not durable
- not production storage
- should preserve metadata exactly

## Metadata Filtering

Storage must support filters when retrieval is scoped.

Common filters:

- user_id
- organization_id
- document_id
- source_type
- tags
- date range
- permissions
- access_scope
- document_visibility

Filter helper:

    def metadata_matches_filters(
        metadata: dict[str, Any],
        filters: dict[str, Any],
    ) -> bool:
        for key, expected_value in filters.items():
            actual_value = metadata.get(key)

            if isinstance(expected_value, list | tuple | set):
                if actual_value not in expected_value:
                    return False
            else:
                if actual_value != expected_value:
                    return False

        return True

Rules:

- apply permission filters before final results
- prefer applying filters before scoring when possible
- do not silently ignore permission filters
- do not retrieve across tenants
- do not rely only on final client-side filtering for sensitive external vector stores unless explicitly accepted

## Deletion and Reindexing

A production storage design must handle updates.

When a document changes:

1. detect text hash change
2. mark old chunks stale or delete them
3. regenerate chunks
4. regenerate contextualized text if enabled
5. regenerate embeddings
6. rebuild or update keyword index
7. clear stale caches
8. record update status

When a document is deleted:

1. delete or soft-delete document record
2. delete associated chunks
3. delete contextualized chunks
4. delete embeddings
5. remove keyword index entries
6. remove cached context
7. remove stale rerank cache
8. preserve audit logs only if policy allows

Do not leave orphaned chunks, embeddings, contextualized text, or keyword entries.

---

## Staleness Detection

Storage should make stale artifacts detectable.

Useful hashes:

- document text hash
- chunk text hash
- indexed text hash
- contextualized text hash
- prompt version
- embedding model
- embedding provider
- indexed_text_type

An artifact is stale when:

- source document text changed
- chunk text changed
- chunking strategy changed
- contextualization prompt changed
- contextualization model changed
- embedding model changed
- embedding provider changed
- indexed_text_type changed

Rules:

- do not reuse stale embeddings silently
- do not reuse stale context silently
- do not return deleted chunks
- do not return chunks from old document versions unless versioning is intentional

## Security Rules

Never store:

- API keys
- access tokens
- plaintext secrets
- unrelated private documents in an unrestricted namespace

Always preserve:

- user_id when user-scoped
- organization_id when organization-scoped
- permissions when permission-scoped
- source metadata needed for auditability
- document_id
- chunk_id

If using external vector databases:

- confirm metadata filtering supports required access controls
- do not rely only on client-side filtering for sensitive data unless explicitly accepted
- namespace or partition tenant data when needed
- verify delete behavior
- verify metadata update behavior
- verify backup and retention policy

## Bad vs Good

Bad:

    chunks = []
    embeddings = []
    # no IDs, no metadata, no persistence

Why bad:

- cannot trace results
- cannot filter by user
- cannot rebuild indexes safely
- cannot persist across restarts
- cannot audit retrieval behavior

Bad:

    vector_db.upsert(text, embedding)

Why bad:

- no chunk_id
- no document_id
- no metadata
- no model tracking
- no indexed_text_type
- no access control
- no source traceability

Bad:

    store_embedding(chunk_id, embedding)

Why bad:

- no model
- no dimension
- no provider
- no metadata
- no indexed text hash
- no stale detection

Good:

    chunk_store.save_chunks(chunks)
    contextualization_store.save_contextualized_chunks(contextualized_chunks)
    embedding_store.save_embeddings(embedded_chunks)
    keyword_index_store.save_keyword_index(keyword_index)

Why good:

- storage responsibilities are separated
- metadata is preserved
- artifacts are traceable
- adapters are replaceable
- indexes can be rebuilt
- stale artifacts can be detected

A storage design is enterprise-ready only if:

- storage backend is selected by the user or explicitly defaulted
- documents have stable IDs
- chunks have stable IDs
- metadata is preserved
- permission metadata is preserved

- original text is separate from contextualized text

- deletion behavior is defined

- stale artifact detection is defined
- API keys are not stored
- storage adapters are replaceable
- tests cover persistence, filters, deletion, reindexing, and model mismatch
