<architecture_principles>
Design the RAG system as layered components.

Use these layers unless the user explicitly requests a simpler design:

1. Ingestion layer

Responsible for:

* loading documents
* extracting text
* preserving source metadata
* validating input
* rejecting empty documents
* preparing text for chunking

2. Chunking layer

Responsible for:

* splitting text into retrieval-safe chunks
* counting or estimating tokens
* preserving metadata
* assigning stable chunk IDs
* handling overlap
* handling oversized chunks

3. Contextualization layer

Responsible for optional contextual retrieval preprocessing.

This layer may:

* generate short LLM-written context for each chunk
* combine the context with the chunk for indexing
* preserve the original chunk text separately
* record contextualization metadata

Important:

Contextual retrieval is normally an indexing-time preprocessing step, not a separate third ranking signal.

Do not treat contextual retrieval as a parallel retriever unless the user explicitly asks for that design.

4. Indexing layer

Responsible for building and storing retrieval indexes.

Possible indexes:

* TF-IDF keyword index
* embedding index
* contextualized TF-IDF index
* contextualized embedding index
* metadata index

5. First-stage retrieval layer

Responsible for producing ranked candidate lists.

Possible first-stage retrievers:

* keyword_search()
* embedding_search()
* metadata_filter()
* recency_filter()
* custom retriever

Each retriever must return a consistent result object.

6. Fusion layer

Responsible for combining multiple ranked candidate lists.

Default recommended fusion:

* Reciprocal Rank Fusion

RRF should be preferred over raw weighted-score fusion unless the user explicitly wants score normalization.

Reason:

TF-IDF scores and cosine similarity scores are not naturally comparable.

7. Reranking layer

Responsible for optional second-stage ranking.

Reranking should run only on a limited candidate pool, not the full corpus.

Recommended pattern:

* retrieve candidates from keyword and embedding retrievers
* fuse with RRF
* take top candidate_pool_size
* rerank candidate pool
* return final top_k

8. Context-building layer

Responsible for:

* selecting final chunks
* respecting max_context_tokens
* preserving source metadata
* including scores and retrieval provenance
* returning context for an LLM

9. Result layer

Responsible for returning:

* query
* final context block
* result list
* result count
* retrieval strategy used
* metadata
* scores
* score provenance
* warnings, if relevant
  </architecture_principles>

<retrieval_strategy_rules>
Support these retrieval strategies when requested.

1. Keyword retrieval

Use TF-IDF unless the user chooses another method.

Keyword retrieval should:

* tokenize text consistently
* ignore or downweight common stop words when appropriate
* score based on term frequency and inverse document frequency
* avoid rewarding naive keyword stuffing
* return ranked results with keyword scores
* preserve metadata

2. Embedding retrieval

Use Voyage AI when requested.

Embedding retrieval should:

* use environment variables for API keys
* isolate Voyage-specific logic in a provider wrapper
* embed documents with document input type when supported
* embed queries with query input type when supported
* validate embedding dimensions
* calculate similarity safely
* return ranked results with similarity scores

3. Hybrid retrieval

Hybrid retrieval means using more than one first-stage retrieval signal.

Typical hybrid setup:

* TF-IDF keyword search
* Voyage embedding search
* RRF fusion
* optional reranking

Do not directly add TF-IDF scores and cosine similarity scores unless a normalization strategy is explicitly defined.

4. Contextual retrieval

Contextual retrieval should:

* run during ingestion
* generate a short context note for each chunk
* use the full document or nearby document context when generating the note
* store original chunk text separately
* index contextualized text
* return original text in the final context unless the user requests contextualized output

5. Metadata filtering

Metadata filtering should be applied before or during retrieval when possible.

Examples:

* restrict by user ID
* restrict by document ID
* restrict by organization ID
* restrict by tag
* restrict by date range
* restrict by permissions

Never ignore permissions metadata if the user says retrieval is permission-scoped.

6. Recency-aware retrieval

Only include recency scoring if the user requests it.

If recency is active, define whether it is:

* a hard filter
* a soft ranking boost
* a separate retrieval signal
* applied before or after fusion
  </retrieval_strategy_rules>

<fusion_policy>
Use Reciprocal Rank Fusion as the default hybrid fusion method when the user allows recommendations.

RRF formula:

score = sum(1 / (k + rank)) across all ranked lists where the chunk appears

Rules:

* Use rank position, not raw TF-IDF or cosine scores.
* Preserve each retriever's original score in score provenance.
* Deduplicate chunks by stable chunk_id.
* Keep contributing retriever names.
* Allow rrf_k to be configurable.
* Default rrf_k may be 60 if the user allows defaults.
* Return fused_score and fused_rank.
* Keep the function swappable so weighted fusion can be added later.

Do not combine raw TF-IDF and cosine scores directly unless the user explicitly asks for weighted-score fusion and a normalization policy is defined.
</fusion_policy>

---

<reranking_policy>
Reranking is optional unless the user explicitly requires it.

Recommended enterprise pattern:

1. Run first-stage retrievers.
2. Fuse ranked lists with RRF.
3. Select top candidate_pool_size.
4. Rerank those candidates.
5. Return final top_k.
6. Build context from final top_k while respecting max_context_tokens.

Rules:

* Do not rerank the full corpus.
* Preserve original retrieval scores.
* Preserve fused rank.
* Add rerank score and rerank rank.
* Make reranking configurable per query when possible.
* Isolate provider-specific rerank API logic in a wrapper.
* Do not require reranking if the user wants a simple local-only system.
  </reranking_policy>

---

<contextual_retrieval_policy>
Contextual retrieval should be treated as an ingestion-time preprocessing step.

Rules:

* Generate a short context note for each chunk.
* The note should explain where the chunk fits in the document.
* The note should be concise.
* Store original chunk text separately from contextualized text.
* Index contextualized text when contextual retrieval is enabled.
* Return original chunk text in the final context unless the user explicitly requests contextualized output.
* Preserve metadata linking contextualized text back to the original chunk.
* Warn the user that contextual retrieval adds ingestion cost and latency.
* Allow contextual retrieval to be disabled.

Do not treat contextual retrieval as a third independent ranking list unless the user explicitly requests experimental parallel contextual scoring.
</contextual_retrieval_policy>

---

<context_building_policy>

* respect max_context_tokens

* include score provenance when requested
* avoid duplicate chunks
* return an empty context safely when no results are found

The context builder should not silently exceed the token budget.
</context_building_policy>
